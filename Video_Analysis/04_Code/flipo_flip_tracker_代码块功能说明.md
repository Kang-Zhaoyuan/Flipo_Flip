# flipo_flip_tracker.py 代码块功能说明

本文用于说明 [flipo_flip_tracker.py](scripts/Video_Analysi_Code/flipo_flip_tracker.py) 中各代码块的职责与执行流程，方便后续维护和二次开发。

## 1. 文件入口与依赖导入

这一部分负责导入标准库、并发工具、路径工具以及 `Video_Analysi_Code` 包下的核心模块。文件顶部的 `__package__` 判断用于支持脚本被直接执行时自动补充搜索路径，避免从命令行运行时出现包导入失败。

这一段的作用是建立运行环境，而不是直接处理业务逻辑。

## 2. 全局常量配置

`DEFAULT_PARALLEL_WORKERS` 定义并行处理的默认最大 worker 数量。

`BLUE_REFERENCE_LEARNING_FILES` 和 `BLUE_REFERENCE_ROUNDS_PER_VIDEO` 是蓝色目标学习的专用配置，前者指定参考视频文件名，后者指定每个视频用于学习的轮数。

这一块的作用是把可调参数集中起来，便于后续统一修改。

## 3. `_silence_worker_streams()`

这个辅助函数在子进程中将 `stdout` 和 `stderr` 重定向到 `os.devnull`，避免多进程并行时控制台输出互相干扰，也减少 Windows 下 spawn 模式的管道异常风险。

如果 `devnull` 打不开，则保持默认输出流不变，保证脚本还能继续运行。

## 4. `_track_one_video()`

这个函数是单个视频的并行处理单元。它会先静默子进程输出，再根据项目根目录和视频路径推导输出 CSV 路径，调用 `run_tracker()` 执行追踪，最后把结果写入 CSV。

返回值包含视频名、颜色名、结果行数和输出文件路径，供父进程汇总进度。

## 5. `_resolve_blue_learning_videos()`

这个函数专门查找蓝色学习用的参考视频，默认从 `02_Videos/Experiments/Blue` 下读取两个指定文件。

它的作用是为蓝色目标提供更稳定的多视频聚合学习数据；如果目录或文件不存在，就返回空列表，供主流程回退到单视频学习。

## 6. `main()` 主流程

这是整个脚本的核心入口，负责串联“读取配置 -> 选择视频 -> 学习颜色参数 -> 建立追踪任务 -> 执行追踪 -> 保存 CSV”的完整流程。

### 6.1 初始化运行参数

先创建 `TrackerParams()` 作为基础追踪参数，再定义支持的视频扩展名集合。随后查找项目根目录、配置文件和颜色配置日志，并加载/保存颜色配置，确保配置文件始终处于可用状态。

### 6.2 选择待处理视频目录

调用交互式目录选择函数获取输入路径，再筛选目录中符合扩展名的视频文件。如果目录里没有视频，脚本直接退出。

### 6.3 为每个视频决定是否学习颜色

脚本逐个读取视频并要求用户指定颜色。若颜色不是 `pink`，则检查该颜色是否已经学习过：

- 如果已经学习过，会询问是否重新学习，并对同颜色结果缓存用户选择，避免重复确认。
- 如果尚未学习过，则默认先学习再追踪。

这一块的作用是把“颜色配置学习”和“视频追踪”串起来，让追踪参数能随实际样本动态更新。

### 6.4 执行颜色学习并写回配置

当需要学习时：

- 若颜色是 `blue`，优先尝试加载专用参考视频，并调用多视频学习函数。
- 如果参考视频缺失，则回退到当前视频的多帧学习。
- 其他颜色则直接对当前视频进行首次学习。

学习成功后，脚本会把结果写回颜色配置文件，并更新内存中的配置载荷，确保后续视频使用最新参数。

### 6.5 构造每个视频的追踪任务

每个视频都会根据颜色配置生成专属的 `TrackerParams`，再通过 `choose_calibration()` 选择对应标定信息，最后加入任务队列 `jobs`。

这一步把“参数准备”和“执行追踪”分离开，便于后面并行调度。

### 6.6 决定单线程还是多线程

脚本根据 CPU 数量、任务数量以及默认并行上限决定 worker 数。如果检测到标准输出被重定向，且当前打算并行执行，则会退回到单 worker 模式，以避免 Windows 下多进程输出管道错误。

### 6.7 单 worker 执行路径

当 worker 数量小于等于 1 时，脚本串行执行每个任务：调用 `run_tracker()`、保存 CSV，并在终端输出当前视频的处理结果。

### 6.8 多 worker 执行路径

当允许并行时，脚本使用 `ProcessPoolExecutor` 提交 `_track_one_video()` 任务，并通过 `as_completed()` 收集结果。每个任务完成后都会打印成功或失败信息，最后汇总成功数和失败数。

这一块是批量处理性能的核心，适合处理多个视频文件。

## 7. `if __name__ == "__main__":` 启动保护

这一段保证脚本既可以被直接运行，也可以作为模块被其他代码导入而不自动执行主流程。

它还捕获 `KeyboardInterrupt`，在用户手动中断时尽量优雅退出，并在必要时把输出流切到 `devnull`，避免关闭管道时出现额外报错。

## 8. 整体执行顺序

整体上，这个脚本的执行顺序可以概括为：

1. 初始化路径、配置和颜色档案。
2. 让用户选择视频文件夹并筛选视频。
3. 为每个视频确定颜色并按需学习颜色参数。
4. 构建追踪任务并选择单线程或多线程执行。
5. 调用追踪管线生成结果 CSV。

## 9. 维护建议

如果后续要扩展新的颜色学习逻辑，优先修改颜色学习相关分支，而不是直接改追踪执行部分。

如果后续要增加新的视频格式，只需要扩展 `video_extensions` 集合即可。

如果要调整批量性能，优先检查 `DEFAULT_PARALLEL_WORKERS` 和多进程输出控制逻辑。
